## Domain Boundaries

```
┌─────────────────────────────────────────────────────────────────────┐
│  USER & WORKSPACE DOMAIN (V2)                                        │
│  users · workspaces · workspace_members · outbox · activity_logs     │
└──────────────────────────┬──────────────────────────────────────────┘
                           │ workspace_id FK (tenant isolation)
┌──────────────────────────▼──────────────────────────────────────────┐
│  BRIEFING DOMAIN (V3)                                                │
│  briefing_sessions · briefing_questions · briefing_answers           │
│  ai_generations · briefing_activity_logs                             │
└──────────────────────────┬──────────────────────────────────────────┘
                           │ briefing_id FK (RESTRICT — cannot delete
                           │ briefing while proposal references it)
┌──────────────────────────▼──────────────────────────────────────────┐
│  PROPOSAL DOMAIN (V4)                                                │
│  proposals · proposal_versions · approval_workflows · approvals      │
└─────────────────────────────────────────────────────────────────────┘
```

## State Machines

### Proposal status
```
DRAFT ──publish──> PUBLISHED ──approve──> APPROVED
                      └────reject──────> REJECTED
```

### ApprovalWorkflow status
```
PENDING ──start──> IN_PROGRESS ──all approved──> APPROVED
                        └──any rejected──────> REJECTED
```

### Approval (individual) status
```
PENDING ──client action──> APPROVED
              └──────────> REJECTED
```
Note: `IN_PROGRESS` was intentionally removed from `approvals.status`.
A client's individual decision is atomic — the intermediate state belongs
to `approval_workflows`, not to the individual record.

## Key Design Decisions

| Decision | Reason |
|----------|--------|
| `client_id` in proposals is NOT a FK | Clients are in a different bounded context; no cross-domain FK to maintain flexibility for future client microservice extraction |
| `proposal_versions` has ON DELETE CASCADE | Versions are subordinate to the proposal aggregate — if proposal is deleted (admin action), history goes too |
| `approval_workflows` has UNIQUE(proposal_id) | Business invariant: exactly 1 workflow per proposal. Enforced at DB level, not just application |
| `briefing_id` ON DELETE RESTRICT | Prevents deleting a briefing that has proposals referencing it — prevents orphaned proposals |
| `proposals.scope_json` is denormalized | Fast reads: latest scope without joining proposal_versions. Kept in sync by application on every scope change |
| `approver_email` is NOT a FK to users | Clients approving proposals are external — they may not have accounts in the system |
| Optimistic locking on proposals | `version BIGINT` with JPA `@Version` prevents concurrent edit conflicts (two users editing same proposal) |
